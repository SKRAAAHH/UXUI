# Відповіді на контрольні та дослідницькі питання

## 1. Контрольні запитання

### Яке призначення Problem Statement?
Чітке формулювання проблеми користувача, яке фокусує команду на головній задачі та відповідає: "Хто, яка проблема і чому її треба вирішити".

### Чим User Persona відрізняється від реального користувача?
**User Persona** — збірний образ ЦА на основі досліджень. **Реальний користувач** — жива людина. Персона — інструмент для проєктування, а не точна копія.

### Для чого потрібні вайрфрейми?
Це схеми інтерфейсу для узгодження структури та логіки сторінки ДО роботи з візуальним дизайном.

### Чому UX-тестування важливе ще до початку програмування?
Бо виправити помилку в дизайні в 100 разів дешевше, ніж після початку розробки.

### Чому скетчі на папері іноді кращі за макети у Figma на початку?
1. Швидкість генерації ідей
2. Дешевизна змін
3. Клієнти менше бояться критикувати "недороблене"

## 2. Питання на дослідження

### Зв'язок UX-дизайну та Agile-розробки

| Agile | UX інтеграція |
|-------|---------------|
| Ітеративність | Дизайн іде паралельно з розробкою, а не наперед |
| Зворотний зв'язок | Тестування в кожному спринті |
| Комунікація | Дизайнер у команді з розробниками |

**Проблема:** Agile фокусується на швидкості, UX потребує часу на дослідження.  
**Рішення:** Спринт 0 або робота дизайнера на 1-2 спринти вперед.

---

### Метод "5 Чому" (користувач видалив додаток)

1. Чому видалив? → Перестав користуватися
2. Чому перестав? → Не мав часу на 30 хв щодня
3. Чому обирав 30 хв? → Додаток рекомендував
4. Чому це стало проблемою? → Нагадування про "борг" тиснули
5. Чому тиснули? → **Першопричина:** Психологічний тиск через жорсткі вимоги та нагадування

### Кейс "Double Diamond" (Starbucks)

| Етап | Дії |
|------|-----|
| **Discover** | Вивчили, що люди не люблять ранкові черги |
| **Define** | "Як прискорити замовлення для ранкових клієнтів?" |
| **Develop** | Ідея мобільного замовлення + прототипи |
| **Deliver** | Запуск Mobile Order & Pay, оновлення за відгуками |
